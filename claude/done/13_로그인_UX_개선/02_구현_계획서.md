## 목표 요약

역할(운영자·강사·수강생)별로 **인증 방식을 분리**하고,  
수강생은 **학원코드 + 이름 선택 + PIN** 방식으로 로그인할 수 있도록 전환한다.  
이 과정에서 로그인 페이지 UI를 재설계하고, 역할별 라우트 보호 구조도 정비한다.

---

## 영향 범위 전체 지도

```
[DB / Prisma]
  Academy        → code 필드 추가
  User           → email nullable 전환
  StudentProfile → studentCode 필드 추가

[인증 / NextAuth]
  auth.ts        → 수강생용 authorize 로직 추가
  auth.config.ts → (변경 없음)
  next-auth.d.ts → (변경 없음)

[미들웨어]
  middleware.ts  → 신규 생성 (현재 없음)
                   역할별 라우트 보호, 미인증 redirect

[로그인 UI — (auth) 그룹]
  /login              → 역할 선택 화면 (신규 재설계)
  /login/staff        → 기존 이메일+비밀번호 (신규 분리, 클린업)
  /login/student      → 3단계 수강생 로그인 (신규)
  (auth)/layout.tsx   → 좌측 패널 유지, 우측 구조 조정

[API 엔드포인트 — 신규]
  GET /api/auth/academy?code=ACAD-1234
  GET /api/auth/students?academyCode=xxx&classId=yyy

[관리자 화면 — 기존 수정]
  /admin/students      → 학번 발급/표시, PIN 초기화 기능 추가
  /admin/students/[id] → 학번, PIN 관리 섹션 추가
  학원 설정 (신규 페이지 또는 대시보드 추가)
    → 학원코드 확인/재생성

[시드 데이터]
  prisma/seed.ts → Academy.code, StudentProfile.studentCode, PIN 해시 반영
```

## 단계별 작업 계획

### Phase 0 — 결정 사항 확정 (코드 없음)

| 항목 | 결정 | 근거 |
|------|------|------|
| User.email nullable | **Yes — `String?`으로 전환** | 수강생은 이메일 없는 게 정상; 가짜 이메일은 기술 부채 |
| studentCode 위치 | **StudentProfile에** | STUDENT 전용 데이터; User에 넣으면 모든 역할에 컬럼 생김 |
| PIN 저장 방식 | **기존 password 컬럼 재활용, bcrypt** | 스키마 변경 최소화 |
| 학원코드 생성 | **자동 생성 (cuid 앞 8자 대문자)** | 예: `ACAD-3F8K`; 원장이 직접 설정하면 중복·혼란 위험 |
| 이름 목록 보안 | **반 단위로만, 이름 끝 한 글자 마스킹** | 예: 김민○; 클래스 정보 없이 이름만 노출 방지 |
| PIN 시도 제한 | **5회 실패 시 10분 잠금** | 4자리 PIN brute-force 방지 |
| 수강생 로그아웃 | **/login으로 (역할 선택 화면)** | staff와 분리 |

### Phase 1 — DB 마이그레이션

**변경 파일:** `prisma/schema.prisma`

```prisma
model Academy {
  // 추가
  code  String  @unique  // 예: "ACAD-3F8K"
}

model User {
  // 변경: String → String?
  email  String?  @unique
}

model StudentProfile {
  // 추가
  studentCode  String?  // 예: "2025-001"
}
```

**주의사항:**
- `User.email`이 `@unique`이므로, null 허용 후 기존 null 값은 unique 제약에서 제외됨 (PostgreSQL은 null 여러 개 허용)
- 기존 ADMIN/TEACHER 레코드는 email 값 유지 — 영향 없음
- 마이그레이션 실행: `npx prisma migrate dev --name add_student_auth_fields`

### Phase 2 — 로그인 UI 재설계

**목표:** 역할 선택 → 경로 분기. 현재 단일 폼을 3개 경로로 분리.

#### 파일 구조 변경
```
src/app/(auth)/
  layout.tsx          → 좌측 패널은 유지, 우측 children 구조 단순화
  login/
    page.tsx          → 역할 선택 화면 (두 개 큰 버튼)
    staff/
      page.tsx        → 이메일+비밀번호 폼 (기존 기능, 클린 재설계)
    student/
      page.tsx        → 3단계 수강생 로그인
```

#### `/login` — 역할 선택 화면
- 두 개의 큰 카드 버튼
  - **원장·강사**: `→ /login/staff`
  - **수강생**: `→ /login/student`
- 데모 빠른입력, 미리보기 링크 **완전 제거** (개발용 잔재)
- 배경 layout 유지

#### `/login/staff` — 직원 로그인
- 이메일 + 비밀번호
- 뒤로가기 링크 (`← 역할 선택으로`)
- 에러 처리 유지
- 비밀번호 분실 안내 (관리자 문의)

#### `/login/student` — 수강생 3단계
- **Step 1**: 학원코드 입력 → 학원명 표시 확인
- **Step 2**: 반 선택 → 이름 목록(마스킹) → 본인 선택
- **Step 3**: PIN 4자리 숫자 키패드
- 진행 인디케이터 (● ● ●)
- 각 단계 뒤로가기 지원

### Phase 3 — 인증 로직 (NextAuth)

**변경 파일:** `src/lib/auth.ts`

```typescript
// authorize 시그니처 변경
authorize({
  type,           // 'staff' | 'student'
  email,          // staff만
  password,       // staff만
  academyCode,    // student만
  studentId,      // student만 (User.id)
  pin,            // student만
})
```

**student 인증 흐름:**
1. `academyCode`로 Academy 조회
2. Academy의 `id`와 `studentId`로 User 조회 (`role === STUDENT` 검증)
3. `bcrypt.compare(pin, user.password)`
4. PIN 시도 횟수 추적 (Redis 없으면 DB 컬럼으로 임시 구현)

**신규 API 엔드포인트:**

`src/app/api/auth/academy/route.ts`
```
GET /api/auth/academy?code=ACAD-3F8K
→ { id, name } 또는 404
```

`src/app/api/auth/students/route.ts`
```
GET /api/auth/students?academyCode=ACAD-3F8K&classId=xxx
→ [{ id, maskedName }]  // "김민○" 형태
```

### Phase 4 — 미들웨어 (라우트 보호)

**신규 파일:** `src/middleware.ts`

현재 라우트 보호가 없음. 각 layout에서 `useSession()`으로 클라이언트 사이드 체크만 함.  
서버사이드 보호가 없어 URL 직접 접근 시 데이터 노출 위험.

```
/admin/*  → ADMIN 역할만 접근
/teacher/* → TEACHER 역할만 접근
/student/* → STUDENT 역할만 접근
/login/*   → 인증된 사용자 접근 시 역할별 홈으로 redirect
```

### Phase 5 — 관리자 화면 업데이트

#### 학원 코드 관리
- `/admin/dashboard` 또는 별도 설정 페이지에 학원코드 표시
- 코드 재생성 버튼 (원장만)

#### 학생 관리 화면 업데이트
- `/admin/students` 목록: 학번 컬럼 추가
- `/admin/students/[id]`: 학번 발급/수정, PIN 초기화 기능
  - PIN 초기화 → 임시 PIN(예: `0000`)으로 설정 후 학생에게 전달
  - 초기화 후 "첫 로그인 시 변경" 플래그 (선택적)

### Phase 6 — 시드 데이터 업데이트

**변경 파일:** `prisma/seed.ts` (존재하면)

- Academy에 `code: 'ACAD-DEMO'` 추가

```

Phase 4 (미들웨어)
  ↓
Phase 5 (관리자 화면)
  ↓
Phase 6 (시드 데이터)
```
