## 2. 마이그레이션 실행 순서

### Step 1: 스키마 검증

```bash
npx prisma validate
```

예상 결과: `✔ Your Prisma schema is valid.`

### Step 2: Prisma Client 생성

```bash
npx prisma generate
```

예상 결과: `✔ Generated Prisma Client to ./node_modules/@prisma/client`

### Step 3: 마이그레이션 생성 + 실행

```bash
npx prisma migrate dev --name init
```

이 명령어가 하는 것:
1. `prisma/migrations/` 폴더에 SQL 파일 생성
2. Supabase DB에 31개 테이블 + 14개 enum 생성
3. `_prisma_migrations` 메타 테이블 생성

예상 결과:
```
✔ Generated migration: 20260410_init
✔ Applied 1 migration
✔ Generated Prisma Client
```

### Step 4: 마이그레이션 확인

```bash
# 생성된 테이블 확인
npx prisma db pull
```

또는 Supabase 대시보드 → Table Editor에서 테이블 목록 확인:
```
Academy, User, Account, Session, VerificationToken,
StudentProfile, ParentContact,
CurriculumClass, Class, ClassTeacher, Enrollment,
Schedule, Lesson,
Attendance, Assignment, Submission, SubmissionHistory,
WeekNote, Payment, Consultation,
Memo, ReportData, Complaint,
ChurnPrediction, CopilotSession, CopilotQuestion,
RecordingSummary, BotQuestion, BotFAQ,
ReviewSummary, AppSetting
```

## 4. 마이그레이션 관리

### 이후 스키마 변경 시

```bash
# 스키마 수정 후
npx prisma migrate dev --name add_some_field

# 프로덕션 배포 시 (안전하게)
npx prisma migrate deploy
```

### 마이그레이션 리셋 (개발 중만)

```bash
# DB 전체 초기화 (데이터 삭제!) + 재마이그레이션 + 재시드
npx prisma migrate reset
```

> ⚠ 프로덕션에서 절대 사용 금지

### 마이그레이션 상태 확인

```bash
npx prisma migrate status
```

---

## 5. 스키마 변경 시 주의사항

| 변경 유형 | 안전도 | 방법 |
|---------|-------|------|
| 필드 추가 (optional) | ✅ 안전 | `migrate dev --name add_field` |
| 필드 추가 (required) | ⚠ 주의 | default 값 필수 또는 기존 데이터 처리 |
| 필드 삭제 | ⚠ 주의 | 데이터 유실 가능 |
| 테이블 삭제 | 🔴 위험 | 데이터 유실 확정 |
| 관계 변경 | ⚠ 주의 | FK 제약 확인 필요 |
| enum 값 추가 | ✅ 안전 | 기존 데이터 영향 없음 |
| enum 값 삭제 | 🔴 위험 | 해당 값 사용 중인 행 처리 필요 |

## 7. 문제 해결

### "Migration failed: relation already exists"
- 이미 테이블이 있는 DB에 migrate 시도한 경우
- 해결: `prisma migrate reset` (개발) 또는 `prisma db push` (기존 데이터 유지)

### "Seed failed: unique constraint violation"
- seed를 2번 실행한 경우 (중복 데이터)
- 해결: `prisma migrate reset` 후 다시 실행

### "P2002: Unique constraint failed"
- seed 데이터의 이메일이나 복합 유니크가 이미 존재
- 해결: seed에서 `upsert` 사용 또는 `deleteMany` 선행
