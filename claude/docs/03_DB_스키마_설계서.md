## 1. ERD 요약

```
Academy (학원)
 ├── User (사용자) ─── role: ADMIN / TEACHER / STUDENT
 │    └── StudentProfile (수강생 추가 정보)
 │         └── ParentContact (학부모 연락처)
 │
 ├── Class (반)
 │    ├── ClassTeacher (반-강사 N:M)
 │    ├── Enrollment (수강 등록 / 반-학생 N:M)
 │    └── Schedule (시간표 - 요일·시간)
 │
 ├── Lesson (수업 - 실제 진행된 날의 수업)
 │    ├── Attendance (출결)
 │    ├── LessonProgress (진도 체크)
 │    ├── CopilotSession (AI 코파일럿 세션)
 │    │    └── CopilotQuestion (질문 + AI 응답)
 │    ├── RecordingSummary (녹음 → 요약)
 │    └── ReviewSummary (복습 요약 발송)
 │
 ├── Assignment (과제)
 │    └── Submission (제출)
 │         └── SubmissionHistory (작성 이력 - 과정 증빙)
 │
 ├── Curriculum (커리큘럼)
 │    └── CurriculumUnit (단원)
 │
 ├── QuestionBot (질문봇 채널)
 │    └── BotQuestion (질문 + AI 응답 + 강사 답변)
 │
 ├── ChurnPrediction (이탈 예측 - 학생별 일별)
 │    └── ChurnEvent (실제 이탈 기록)
 │
 ├── Complaint (민원 기록)
 │
 ├── Payment (수강료)
 │
 └── MakeupClass (보강 수업)
```

## 2. 테이블 설계

### 2.1 인증 · 사용자

#### User
시스템 전체 사용자. role로 강사·수강생·운영자를 구분한다.

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | String (cuid) | PK |
| email | String | 유니크, 로그인 ID |
| password | String | bcrypt 해시 |
| name | String | 표시 이름 |
| role | Enum | ADMIN / TEACHER / STUDENT |
| academyId | String | FK → Academy |
| isActive | Boolean | 비활성화 여부 (default: true) |
| createdAt | DateTime | |
| updatedAt | DateTime | |

#### Academy
멀티 테넌트 지원을 위한 학원 단위.

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | String (cuid) | PK |
| name | String | 학원명 |
| phone | String? | 대표 연락처 |
| address | String? | 주소 |
| createdAt | DateTime | |

#### StudentProfile
STUDENT role 사용자의 추가 정보.

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | String (cuid) | PK |
| userId | String | FK → User (유니크) |
| grade | String? | 학년 (초등 4학년 등) |
| school | String? | 학교명 |
| memo | String? | 특이사항 메모 |

#### ParentContact
학부모 연락처. 학생 1명당 복수 가능 (부·모 등).

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | String (cuid) | PK |
| studentId | String | FK → User (STUDENT) |
| name | String | 학부모 이름 |
| phone | String | 연락처 (암호화 저장) |
| relation | String? | 관계 (부/모/기타) |
| isMain | Boolean | 주 연락처 여부 |

### 2.2 학원 · 반 · 시간표

#### Class
반 단위. 과목·레벨·정원 등 기본 정보.

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | String (cuid) | PK |
| academyId | String | FK → Academy |
| name | String | 반 이름 (화-초4 친절한 설명) |
| subject | String | 과목 |
| level | String? | 레벨 (입문 / 기초 / 심화) |
| capacity | Int | 정원 |
| curriculumId | String? | FK → Curriculum |
| isActive | Boolean | 운영 여부 |
| createdAt | DateTime | |

#### ClassTeacher
반-강사 N:M 관계. 한 반에 보조 강사 추가 가능.

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | String (cuid) | PK |
| classId | String | FK → Class |
| teacherId | String | FK → User (TEACHER) |
| isPrimary | Boolean | 주 담당 강사 여부 |
| assignedAt | DateTime | |

#### Enrollment
학생-반 수강 등록. 이동 이력 보존을 위해 soft delete.

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | String (cuid) | PK |
| classId | String | FK → Class |
| studentId | String | FK → User (STUDENT) |
| enrolledAt | DateTime | 등록일 |
| leftAt | DateTime? | 퇴반일 (null이면 현재 수강 중) |
| leaveReason | String? | 퇴반 사유 |

#### Schedule
반의 반복 시간표 (요일·시간 기준).

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | String (cuid) | PK |
| classId | String | FK → Class |
| dayOfWeek | Int | 0=월 ~ 5=토 |
| startTime | String | "15:00" 형식 |
| endTime | String | "17:00" 형식 |
| room | String? | 강의실 |

#### Lesson
실제 진행된 수업 인스턴스. Schedule 기반으로 생성.

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | String (cuid) | PK |
| classId | String | FK → Class |
| scheduleId | String | FK → Schedule |
| date | Date | 수업 날짜 |
| topic | String? | 오늘 수업 주제 |
| note | String? | 강사 메모 |
| status | Enum | SCHEDULED / COMPLETED / CANCELLED |
| createdAt | DateTime | |

### 2.9 AI — 수업 코파일럿

**aiResponse Json 구조:**

### 2.13 AI — 이탈 예측

#### ChurnPrediction
학생별 이탈 위험도 점수. 매일 자동 계산.

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | String (cuid) | PK |
| studentId | String | FK → User (STUDENT) |
| classId | String | FK → Class |
| calculatedAt | Date | 계산 날짜 |
| score | Int | 위험도 점수 (0~100) |
| level | Enum | SAFE / WARNING / DANGER |
| factors | Json | 점수 구성 요인 상세 |
| isNotified | Boolean | 운영자 알림 발송 여부 |

**factors Json 구조:**
```json
{
  "attendance": { "score": 25, "detail": "최근 2회 연속 결석" },
  "homework":   { "score": 20, "detail": "최근 3회 미제출" },
  "access":     { "score": 15, "detail": "1주 미접속" },
  "questions":  { "score": 0,  "detail": "정상" }
}
```

**유니크 제약:** `(studentId, classId, calculatedAt)`

## 5. 설계 결정 사항

### Soft Delete 사용 이유 (Enrollment)
학생이 반을 이동해도 이전 수강 이력이 보존되어야 이탈 분석, 성장 추이 파악이 가능하다.
`leftAt`이 null이면 현재 수강 중, 값이 있으면 퇴반 처리.

### AI 응답 Json 컬럼 사용 이유
`CopilotQuestion.aiResponse`, `ChurnPrediction.factors` 등은 구조가 고정되지 않고 AI 기능 고도화에 따라 변할 수 있어 Json으로 유연하게 저장. 핵심 필터링 필드만 별도 컬럼 분리.

### cuid() 사용 이유
UUID보다 짧고 정렬 가능성이 있어 URL 노출 시 가독성이 낫고 인덱스 성능도 양호.

### 멀티 테넌트 구조
`Academy` 최상위 → 모든 데이터에 `academyId` 연결. 단일 DB로 여러 학원 데이터를 격리. 추후 학원별 요금제 적용 가능.

### 개인정보 처리
`ParentContact.phone`은 애플리케이션 레벨에서 암호화 후 저장 (AES-256). DB 컬럼 타입은 String 유지, Prisma middleware에서 encrypt/decrypt 처리.
