## 1. 페이지 상태 흐름

모든 데이터 페이지는 아래 5가지 상태를 순환합니다.

```
[초기 로딩] ──→ [정상 표시] ──→ [데이터 새로고침]
     │               │                │
     │               ▼                │
     │          [빈 상태]              │
     │               │                │
     ▼               ▼                ▼
[에러 상태] ←── [에러 발생] ←── [에러 발생]
     │
     └── [다시 시도] ──→ [초기 로딩]
```

| 상태 | 조건 | 표시 방식 |
|------|------|---------|
| **초기 로딩** | 첫 데이터 fetch 중 | 스켈레톤 UI |
| **정상 표시** | 데이터 존재 | 실제 콘텐츠 |
| **빈 상태** | 데이터 0건 | EmptyState 컴포넌트 |
| **에러 상태** | API 실패 | 에러 메시지 + [다시 시도] |
| **새로고침** | SWR revalidation 중 | 기존 데이터 유지 + 조용한 로딩 표시 |

## 4. 에러 처리 전략

### 4.1 HTTP 에러 코드별 처리

| 에러 코드 | 상황 | 표시 방식 | 사용자 액션 |
|---------|------|---------|-----------|
| **400** | 유효성 검사 실패 | 인라인 에러 (필드 하단 빨간 텍스트) | 입력값 수정 |
| **401** | 인증 만료 / 미인증 | `/login`으로 리다이렉트 | 재로그인 |
| **403** | 권한 없음 | 전체 페이지 에러 화면 | [홈으로] |
| **404** | 리소스 미존재 | 전체 페이지 에러 화면 | [뒤로가기] |
| **409** | 충돌 (동시 수정) | 토스트 알림 | [새로고침] |
| **429** | 요청 과다 | 토스트 알림 | 대기 후 재시도 |
| **500** | 서버 오류 | 인라인 에러 또는 전체 에러 | [다시 시도] |
| **네트워크** | 연결 끊김 | 상단 배너 (persistent) | [다시 시도] |

### 4.2 에러 표시 형태

#### 인라인 에러 (폼 필드)

```
┌───────────────────────────┐
│  📧  user@example          │   ← border: red-500
└───────────────────────────┘
 이메일 형식이 올바르지 않습니다    ← text-red-500, 12px
```

#### 폼 상단 에러 배너

```
┌───────────────────────────────────────────┐
│  ❌  이메일 또는 비밀번호가 올바르지 않습니다  │   ← bg-red-50, text-red-700
└───────────────────────────────────────────┘
```

#### 토스트 에러

```
┌──────────────────────────────────┐
│  ❌  저장에 실패했습니다.       [×]  │   ← 우측 상단, 3초 자동 닫힘
└──────────────────────────────────┘
```

#### 전체 페이지 에러

```
┌─────────────────────────────────┐
│                                 │
│          ⚠                      │
│                                 │
│   요청하신 페이지를 찾을 수 없습니다  │
│                                 │
│     [홈으로]   [뒤로가기]         │
│                                 │
└─────────────────────────────────┘
```

#### 네트워크 오류 배너 (persistent)

```
┌─────────────────────────────────────────────────────┐
│  ⚠  네트워크 연결을 확인해주세요              [다시 시도]  │
└─────────────────────────────────────────────────────┘
  ← 페이지 상단 고정, 연결 복구 시 자동 사라짐
```

### 4.3 SSE 스트리밍 에러 (AI 코파일럿)

```
정상 스트리밍 중:
  ⏳ 생성 중... 1.2s
  [카드 내용이 실시간으로 채워짐]

연결 끊김:
  ┌────────────────────────────────────────┐
  │  ⚠ AI 연결이 끊겼습니다                  │
  │     [재연결]  [질문 다시 보내기]           │
  └────────────────────────────────────────┘

타임아웃 (10초 무응답):
  ┌────────────────────────────────────────┐
  │  ⏱ 응답이 너무 오래 걸리고 있습니다       │
  │     [다시 시도]  [취소]                  │
  └────────────────────────────────────────┘
```

## 5. 낙관적 업데이트 (Optimistic Update)

즉각적인 피드백이 필요한 UI에 적용합니다.

### 5.1 적용 대상

| 화면 | 액션 | 낙관적 처리 | 실패 시 롤백 |
|------|------|-----------|------------|
| **출결 관리** | 상태 버튼 클릭 | 즉시 버튼 색상 변경 + 통계 갱신 | 이전 상태 복원 + 에러 토스트 |
| **과제 자동저장** | 5분 자동 | "자동저장 완료" 표시 | "⚠ 저장 실패" 빨간 텍스트 |
| **질문 전송** | [질문 보내기] | 즉시 질문 카드 추가 (대기 중) | 카드 제거 + 에러 토스트 |
| **알림 읽음** | 알림 클릭 | 즉시 배지 숫자 감소 | 숫자 복원 |
| **과제 제출** | [제출하기] | - (확인 모달 거치므로 낙관적 불필요) | - |

### 5.2 낙관적 업데이트 흐름

```
사용자 액션
    │
    ├── 1. UI 즉시 변경 (mutate local state)
    │
    ├── 2. API 호출 (백그라운드)
    │       │
    │       ├── 성공 → 서버 데이터로 로컬 상태 동기화
    │       │
    │       └── 실패 → 이전 상태로 롤백
    │                   + 에러 토스트 표시
    │
    └── 3. SWR revalidation (서버 데이터 재확인)
```

### 5.3 자동저장 상태 표시 (과제 작성)

```
저장 성공:  342자 · ✅ 자동저장 완료 · 마지막 저장 20:30
저장 중:    342자 · ⏳ 저장 중...
저장 실패:  342자 · ⚠ 저장 실패 · [수동 저장]
수정됨:    345자 · 📝 수정됨 (저장 안 됨)
```

## 6. 로딩 상태 세분화

### 6.1 초기 로딩 vs 갱신 로딩

| 상황 | 표시 |
|------|------|
| **첫 로딩** (isLoading=true, data=undefined) | 스켈레톤 UI 전체 표시 |
| **갱신 중** (isValidating=true, data=존재) | 기존 데이터 유지 + 조용한 인디케이터 |
| **에러 후 재시도** | [다시 시도] 버튼 스피너 |

### 6.2 버튼 로딩 상태

```
기본:     [  등록  ]
로딩:     [ ⏳ 등록 ]   ← 스피너 + 텍스트, opacity 0.7, 클릭 비활성
비활성:   [  등록  ]   ← opacity 0.5, cursor not-allowed
```

### 6.3 페이지 전환 로딩

Next.js App Router에서 라우트 전환 시:
- 상단에 얇은 프로그레스 바 표시 (sky-500, h:2px)
- `nprogress` 또는 커스텀 구현

## 7. 데이터 동기화 전략

### 7.1 SWR 설정

| 화면 | revalidateOnFocus | refreshInterval | 이유 |
|------|:-:|:-:|------|
| 대시보드 | ✅ | 5분 | 실시간성 필요 |
| 학생/반 목록 | ✅ | - | 포커스 시 최신 데이터 |
| 출결 관리 | ✅ | - | 다른 강사 입력 반영 |
| 코파일럿 | ❌ | - | SSE로 실시간 (SWR 불필요) |
| 수강생 홈 | ✅ | 10분 | 적당한 갱신 |
| 복습 자료 | ✅ | - | 포커스 시만 |

### 7.2 캐시 무효화

특정 액션 후 관련 캐시를 무효화:

| 액션 | 무효화 대상 |
|------|-----------|
| 학생 등록 | `/api/users?role=STUDENT` 캐시 |
| 출결 입력 | `/api/attendance` + `/api/attendance/stats` |
| 과제 등록 | `/api/assignments` 캐시 |
| 과제 제출 | `/api/assignments/[id]` 캐시 |
| 납부 처리 | `/api/payments` 캐시 |
| 질문 전송 | `/api/ai/bot/questions` 캐시 |
