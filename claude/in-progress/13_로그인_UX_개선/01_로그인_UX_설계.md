# 로그인 UX 개선 설계

## 배경

현재 단일 로그인 화면(`/login`)에서 운영자·강사·수강생 모두 이메일+비밀번호로 로그인.
실제 학원 운영 환경에서는 역할별 인증 방식이 달라야 자연스럽고 편리함.

---

## 역할별 인증 방식 분석

### 운영자 (원장)
- 소수 (학원당 1~2명)
- 결제·개인정보 등 민감 데이터 접근
- **방식: 이메일 + 비밀번호 유지** → 보안상 가장 적합
- 추가 고려: 향후 2FA(이메일 OTP) 가능

### 강사
- 학원에서 계정 발급받는 구조
- 이메일 있지만 학원마다 별도 계정 관리 번거로움
- 강사가 여러 학원 소속일 수도 있음
- **방식 A (현재 유지):** 이메일 + 비밀번호 — 관리자가 계정 발급
- **방식 B (개선):** 학원 코드 + 강사 아이디 + 비밀번호
- → **단기: 방식 A 유지**, 추후 방식 B 전환 고려

### 수강생 (핵심 개선 대상)
- 초중고생 → 이메일 없거나 잘 안 씀
- 비밀번호 분실 잦음 (원장/강사가 자주 리셋해줘야 함)
- 학부모가 대신 로그인하는 경우 있음
- 입력 실수 최소화 필요

**방식 비교:**

| 방식 | 편의성 | 보안 | 관리 부담 |
|------|--------|------|-----------|
| 이메일+비밀번호 (현재) | 낮음 | 높음 | 높음 |
| 학원코드+이름+PIN | 높음 | 중간 | 낮음 |
| 학원코드+학번+생년월일 | 높음 | 중간 | 낮음 |
| QR코드 스캔 | 매우 높음 | 낮음 | 낮음 |
| 전화번호+OTP | 높음 | 높음 | 중간 |

**추천: 학원코드 + 이름 선택 + PIN 4자리**
- 클래스팅, 아이스크림홈런 등 유사 서비스 채택 방식
- 원장이 학생 등록 시 PIN 발급 → 학생이 첫 로그인 후 변경

## 제안하는 전체 로그인 플로우

```
시작 화면 (/)
│
├── [원장 · 강사 로그인]              [수강생 로그인]
│         ↓                                  ↓
│   이메일 입력                      학원 코드 입력
│   비밀번호 입력                    (예: ACAD-1234)
│         ↓                                  ↓
│    역할 자동 분기                   반 선택 또는
│  (ADMIN → /admin/dashboard)        이름으로 검색
│  (TEACHER → /teacher/dashboard)            ↓
│                                    본인 이름 선택
│                                            ↓
│                                    PIN 4자리 입력
│                                            ↓
│                                  /student/home 이동
```

## DB 스키마 변경 사항

### Academy 테이블 — `code` 필드 추가
```prisma
model Academy {
  // 기존 필드들...
  code  String  @unique  // 예: "ACAD-1234", 학원 접속 코드
}
```

### User 테이블 — 수강생 관련 필드 추가
```prisma
model User {
  // 기존 필드들...
  studentCode  String?  // 학번 (수강생만 사용, 예: "2025-001")
}
```

### PIN 처리
- 별도 필드 없이 기존 `password` 필드 그대로 사용
- PIN → bcrypt 해시 후 저장 (4자리도 bcrypt 가능)
- 기존 이메일+비밀번호 방식과 동일한 컬럼 재활용

## API 변경 사항

### 신규 엔드포인트
```
GET /api/auth/academy?code=ACAD-1234
→ 학원 존재 여부 확인 + 학원명 반환

GET /api/auth/students?academyCode=ACAD-1234&classId=xxx
→ 해당 학원 수강생 목록 반환 (이름만, 보안상 최소화)
```

### NextAuth 변경
- Credentials provider에 수강생용 인증 로직 추가
- `identifier` 타입 구분: `email` vs `studentCode`

```typescript
// 현재
authorize({ email, password })

// 변경 후
authorize({ type, email, password, academyCode, studentCode, pin })
```

## UI 구성 (페이지별)

### `/login` — 역할 선택 화면
- 두 개의 큰 버튼: [원장·강사 로그인] / [수강생 로그인]
- 각각 `/login/staff`, `/login/student`로 이동

### `/login/staff` — 기존 이메일 로그인
- 현재 폼 그대로 유지

### `/login/student` — 3단계 스텝 UI
1. **학원 코드 입력** → 학원명 확인
2. **반 선택 → 이름 선택** (드롭다운 또는 검색)
3. **PIN 4자리 입력** (숫자 키패드 UI)

## 구현 우선순위

| 순서 | 작업 | 난이도 |
|------|------|--------|
| 1 | Academy 테이블에 `code` 필드 추가 (마이그레이션) | 낮음 |
| 2 | User 테이블에 `studentCode` 필드 추가 | 낮음 |
| 3 | `/api/auth/academy`, `/api/auth/students` 엔드포인트 | 중간 |
| 4 | NextAuth Credentials 수강생 인증 로직 추가 | 중간 |
| 5 | `/login` 역할 선택 UI | 낮음 |
| 6 | `/login/student` 3단계 UI | 중간 |
| 7 | 관리자 화면에서 학원코드·학번 발급/관리 | 중간 |
| 8 | 씨드 데이터 학원코드·PIN 반영 | 낮음 |
